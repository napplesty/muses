Muses is a header-only toolkit of concurrency and networking primitives — a per-allocation memory pool, an async logger, and a reactor-based HTTP server. Tested under ASan + UBSan.

Why muses

◈

Memory pool

Size-classed free-lists with true per-allocation reference counting. PoolPtr<T> RAII handle, magic-guarded against double-free.

∿

Async logging

Bounded MPMC ring with DropOldest — the hot path never blocks. The consumer can no longer deadlock.

⟳

Single-reactor server

One I/O thread + a worker pool, handoff through a single bounded outbox — the only shared channel, so no data races.

⬡

Poller abstraction

Poller unifies kqueue and epoll behind one interface, with a cross-thread wakeup().

▤

Bounded queue

DropNew / DropOldest / Block overflow policies, timeout-aware pop, clean shutdown.

⌁

HTTP/1.1 handler

CRLF-correct, binary-safe, path-traversal-proof, with keep-alive support. Powered by std::format.

A server in ten lines

// examples/http_server.cpp
muses::TCPListener listener("127.0.0.1", 8864);
muses::Reactor reactor(listener.get_listener().value(),
    [](std::string req) -> muses::HandlerResult {
        auto hr = muses::HttpContext::handle_request(req);
        return {std::move(hr.response), hr.keep_alive};
    }, /*workers=*/4);
reactor.start();
